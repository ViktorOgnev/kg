# Mermaid стиль для «Карты гипотез» (кратко)

Базовая конвенция
- Визуальный порядок уровней должен читаться слева направо: цель, затем субъекты, затем гипотезы, затем задачи.
- Если Mermaid требует инверсии направления графа, допустимо использовать `flowchart RL`, но итоговая картинка всё равно должна читаться как «цель слева, развитие вправо».
- Для больших карт предпочитать `layout: elk`, а не стандартный `dagre`: он обычно лучше раскладывает многоуровневые схемы с большим количеством узлов.

Рекомендуемые полосы (bands)
- Goal band: цель и связанные с ней метрики.
- Metric band: при сложной карте метрики и балансирующие метрики вынести на отдельный уровень между целью и субъектами.
- Subject band: субъекты, включая negative subjects.
- Hypothesis band: гипотезы.
- Task band: задачи; если есть `status`/`lane`, допустима доска задач вместо простого списка.
- Context band: blockers и notes, по возможности внизу.

Связи
- Базовая causal chain: `task --> hypothesis --> subject --> goal`.
- Метрики обычно связываются как `metric --> goal`.
- Если стрелок на метрики становится слишком много, влияние гипотез на метрики лучше показывать тэгами/подписями внутри hypothesis node, а не отдельным лесом стрелок.
- Subject-to-metric arrows лучше не рисовать: их влияние считать неявным.

Когда использовать расширенную топологию
- Есть отдельный уровень метрик и балансирующих метрик.
- Есть negative subjects.
- Есть много задач и их нужно группировать по lane/status.
- Обычная causal chain уже плохо читается из-за количества стрелок.
